### The one move (highest leverage)
Add an execution-fidelity self-check to the orchestrator's "after a wave returns" step: before writing run-state, the orchestrator asserts that every dispatch it just marked as belonging to one parallel wave was in fact emitted in a single assistant message, and records a `dispatch_mode: parallel|serial` fact per wave in run-state. This is the one move because it attacks the root cause of the worst defect this run produced. Waves 2 and 3 were serialized one dispatch per message, which `references/parallel-dispatch.md` line 3 calls out in the engine's own words as the silent failure: "the agents run in parallel silently becomes the agents run one at a time." It is the *silent* part that makes it lethal: run-state.json records waves 2 and 3 as parallel (the DAG is correct), so the state file looks perfect while execution was serial. No content gate can see this, because the auditor gates artifacts, not the dispatch stream. Fixing the enum (below) is a clean one-line schema edit, but the parallelism gap is the engine failing to deliver the one thing it exists to do, on the one run that was supposed to prove it. A recorded `dispatch_mode` turns an invisible regression into a visible, gradeable fact the retrospective can catch every run. It beats "just remind the orchestrator to batch" because reminders decay; a recorded assertion does not.

### Risks and gaps
- **Recorded-parallel vs executed-serial divergence (highest risk).** run-state.json waves 2-3 are recorded as parallel; the run brief states they were emitted serially. The state file cannot be trusted as evidence the engine ran parallel, because it records intent, not execution. Evidence: run-state.json L40-53 (parallel DAG) vs the dispatch-stream reality in the brief; the failure mode is named in parallel-dispatch.md L3.
- **Verdict enum contradiction is a live spec defect.** protocols/run-state.schema.json L98 enum = PASS/WARN/BLOCK; SKILL.md L78 and wave-planning.md L58/L60 instruct ship/iterate/block. Confirmed contradiction in the engine's own files, not a test artifact. The ITERATE->BLOCK map (run-state.json L53, 3.auditor.gate L47) is the symptom.
- **No execution-mechanics gate exists anywhere.** The auditor gates content (math, consistency, safety floor, completeness, grounding). It never checks how a wave was dispatched. So the parallelism failure is structurally uncatchable today. This is the root gap behind the one move.
- **Self-improvement loop did not close.** run-state.learning is empty and retrospective_path is null; DoD item 6 (run report + retrospective + ledger) is pending. The engine has not yet demonstrated the slow loop end to end, only the fast loop. The golden run is not fully golden until the ledger is written.

The auditor caught and flagged its own earlier number rather than burying it. This is the gate working, not a defect, but it argues for testers computing contrast from hex rather than eyeballing.
